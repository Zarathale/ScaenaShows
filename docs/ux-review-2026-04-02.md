# ScaenaShows — UX Review
**Date:** 2026-04-02
**Scope:** Full plugin, admin/ops perspective (Alan + Smitty)
**Focus:** Phase 1 setup interface, Phase 2 in-game cue authoring, systemic workflow friction
**Status:** Working document — use to drive next design sprint

---

## Context and Framing

ScaenaShows is an ops-only system on a private whitelist server. The audience (players) experiences finished shows; the authors (Alan, Smitty) design, configure, and rehearse them. This review evaluates the **author experience** — the path from "I have a show concept" to "I can run this show in-game and it feels right."

The engine itself is sound. The event executor system, runtime orchestration, and YAML schema are well-architected. The problem is almost entirely at the **authoring surface**: how an admin discovers what needs to be done, makes decisions, edits content, and confirms the result — all while staying in the creative flow of building a show.

Two problems stand out as foundational. Everything else is downstream of these.

---

## The Two Foundational Problems

### Problem 1: No Phase 1 Interface

When you open a show for the first time — or return to one mid-development — there's no single place that answers: *"What does this show need, and what state is each piece in?"*

The prompt-book captures this information perfectly at the data level: scenes, marks, params, readiness items, TBD values. But there's no in-game surface that reads it and presents it as a **show status dashboard**. The closest thing is the tech panel (`/scaena tech panel`), which shows department toggles and a marks row — but it's a *working panel for an active session*, not a *status overview you consult to know what's left.*

The result: to understand your show's current state, you have to open the YAML file, read the `readiness.open` block, cross-reference the scene list, check the scout_captures file for captured marks, and mentally assemble the picture. That's friction that breaks the creative loop every time.

**What's missing:** A command that says *"Here's your show. Here's what's set. Here's what isn't. Here's what to do next."*

### Problem 2: Phase 2 Requires Leaving the Game

The current architecture assumes that all cue-level content — the show's timeline, the events, the cue library — is authored externally in YAML files. To change what a show does, you edit YAML, run `/show reload`, and check the result in-game. There is no in-game interface for editing, adding, or removing cues on a show's timeline.

This is the most significant design gap. It means:

- The feedback loop (edit → see result) requires a context switch out of Minecraft
- The show's "skeleton" (rough timeline from YAML) has no in-game representation the author can interact with
- There's no way to browse the cue library, select a cue, and drop it onto the timeline without writing YAML
- Parameters within cue events (sound pitch, particle count, firework preset) can't be tuned in-game

For a creative tool meant to be used *live, during creative flow*, forcing YAML authorship for every timeline change is a fundamental design misalignment. The Phase 2 interface — browsing the timeline, placing cues, adjusting event parameters — needs to exist in-game.

## Current State Analysis

### The Three Workflows (as designed)

The plugin has three distinct operational modes with separate command namespaces, session managers, and state systems:

**Scouting** (`/scaena scout *`)
Captures spatial positions (marks) for scenes. Walk to a spot, run `/scaena set <code>`, position is saved. Has its own session (`ScoutSession`), scoreboard, TextDisplay markers, and output file (`scout_captures/`).

**Tech Rehearsal** (`/scaena tech *`)
Configures a show in-game: materializes entities, places set blocks, toggles departments, captures marks (again), edits parameters. Has its own session (`TechSession`), sidebar display, actionbar task, hotbar items, and chat panel.

**Show Playback** (`/show *`)
Runs finished shows. Clean separation — takes a loaded show and executes it. This workflow is well-designed and doesn't have significant UX problems.

### The Fragmented Setup Path

The first two workflows — scouting and tech — are conceptually part of the same activity: *setting up a show*. But they're implemented as completely separate systems. You must scout positions in one session, save to a captures file, exit, then enter tech to load those captures and continue work. They have different commands, different state, different visual feedback, and different save flows.

This split reflects an internal architectural distinction (position capture vs. parameter editing), not a user-meaningful one. A show director thinking about their show doesn't think "now I'm scouting, now I'm in tech" — they think "I'm setting up this scene."

### What the Current UI Surfaces

These are the in-game UI elements that currently exist:

**TechSidebarDisplay** — scoreboard on the right side of screen. Shows: show title, scene number/label, casting entries with capture status, department toggles (✓ / — / ·). This is good information, presented clearly. The main gap is that it only shows the current scene, not the show's overall completion state.

**TechPanelBuilder** — clickable chat panel sent on session start and via `/scaena tech panel`. Shows department toggles, mark names as clickable links, and action buttons (Params, Save, Discard, Exit). This is functional but fragile: it scrolls off screen with any chat activity and must be manually re-summoned. Marks are listed by name as inline links — you click to enter capture mode for that mark — but there's no status symbol on the panel itself.

**TechActionbarTask** — 5-tick repeating display. Shows context-appropriate text: session state normally, live coordinates in capture mode, param value in scroll mode. Well-designed; the right pattern for transient state.

**ScoutSession markers** — TextDisplay entities in-world at captured positions. Only available in scout mode. These are the right pattern for spatial feedback but should be present in tech mode too.

**Mark list panel** — sent by `/scaena tech marklist`. Shows each mark with status symbols (✓ captured, ⚠ missing, ✎ modified). This is the closest thing to a per-scene status view — but it's a separate panel you have to summon.

## Full UX Problem Inventory

### 1. No Show-Level Status View [Critical]

There is no command that shows the complete configuration state of a show — across all scenes, all marks, all departments, and all open readiness items. You can see one scene at a time in the sidebar, and summon mark/param panels for the current scene, but there's no holistic view.

The prompt-book's `readiness.open` block is exactly the right data source for this — it's where TBD items are tracked — but nothing reads it and presents it as a checklist.

**Where this breaks:** Opening a show after a few days away and needing to remember what still needs to be done.

### 2. No In-Game Timeline or Cue Authoring [Critical]

The show's timeline (`timeline:` in the show YAML) is completely invisible in-game. You can't see what cues are scheduled, when they fire, or what they do. You can't add a cue, remove one, or change a cue's `at:` value without editing YAML externally.

The cue library (`cues/*.yml`, 60+ entries) exists and is loaded by the engine, but there's no in-game browser for it. You have to know a cue's ID by memory or by looking at files.

**Where this breaks:** Every iterative step of show authoring. "I want to add a firework burst at the 30-second mark" → exit game → open YAML → add event → save → `/show reload` → check result in-game. This loop is too expensive for creative work.

### 3. Scout and Tech Are Two Sessions for One Task [High]

Position capture and scene configuration are both part of "setting up a scene." Splitting them into separate tools with separate sessions, separate commands, and separate save flows doubles the cognitive overhead and the friction of context-switching.

The duplication is also architectural: both `ScoutSession` and `TechSession` manage captures, both have sidebar scoreboards, both manage TextDisplay markers. The scouting workflow (`/scaena scout`) partially overlaps with tech capture mode (`/scaena tech capture <mark>`) — they accomplish the same outcome via different interfaces.

**Where this breaks:** When you discover mid-tech that a mark position needs to change. You either use tech capture mode (which requires knowing the exact mark name) or dismiss, re-enter scout, recapture, re-enter tech.

### 4. The Chat Panel Is Fragile [High]

`TechPanelBuilder` produces a good panel — the information hierarchy is right, the clickable buttons work. But chat panels are fundamentally fragile:

- Any server output (another player chatting, a command confirmation, server messages) pushes the panel off screen
- There's no automatic refresh when state changes — you must run `/scaena tech panel` to re-summon it
- Long sessions accumulate multiple stale panels in scroll history

The panel also doesn't show mark capture status inline. A mark listed in the marks row looks the same whether it's captured or missing. The detail (✓/⚠/✎) is only visible in the mark list panel, which is a separate invocation.

**Where this breaks:** Any work session longer than a few minutes, or when playing on a server with active chat.

### 5. Parameter Editing Requires Knowing Param Names [High]

To edit a parameter, you run `/scaena tech focusparam <paramName>` — but you have to know the parameter's name. The param panel (`/scaena tech params`) shows the list, but it's a second step. If you've scrolled the panel off screen and don't remember the param names, you're back to the YAML file.

Parameter types are also implicit. `ParamSpec` has a `type` field (NUMERIC | TEXT) and step/min/max constraints, but the actionbar in param scroll mode shows the value and controls without labeling what the parameter is *for*. A notes field exists in the YAML but it's not surfaced in the UI.

**Where this breaks:** Working on a show you haven't touched in a week. Every param feels opaque.

### 6. No Spatial Feedback in Tech Mode [Medium]

In scout mode, TextDisplay markers appear at captured positions (`markersVisible` in `ScoutSession`). In tech mode, markers also appear after capture — but there's no "show me all marks" overview for the current scene. Uncaptured marks have no in-world indicator (no floating `?` to show where they're *expected* to be, based on prior captures or default offsets).

Show marks are `x/z offsets from anchor` — there's no visual indicator of where the anchor is, or where offset positions would fall.

**Where this breaks:** Spatial reasoning about a show while standing in the actual build location.

### 7. Command Discoverability Is Low [Medium]

The `/scaena` namespace has 20+ subcommands across two nested levels. Tab completion exists but doesn't explain context — it offers all valid next tokens regardless of session state. Outside of an active session, `capture` and `toggle` are visible in tab completion even though they'd fail without a session.

There's no help surface that adapts to state: "you're in a tech session — here's what you can do right now." The `/scaena` bare command returns a help listing, but it's a static dump, not contextual.

**Where this breaks:** Learning the system, or returning after a break. Also blocks Smitty from using the tools independently without a reference guide.

### 8. Build Mode Has No Visual Boundary [Medium]

Entering build mode (switching to SURVIVAL, tracking block changes) has no spatial feedback about the current bounding box. You set bbox corners with `/scaena tech bbox min/max` but you can't see the box. If the corners are set wrong — or not set at all — you don't know until you look at the saved values.

**Where this breaks:** Set-dressing work. You're placing blocks, you think you're inside the bbox, but you're not — and those blocks won't be managed correctly.

### 9. Capture Mode Entry/Exit Is Implicit [Low]

Entering capture mode (`/scaena tech capture <mark>`) changes the actionbar to show live coordinates and switches the slot-8 hotbar item to a "capture" action. But there's no audio cue, no screen flash, no unambiguous "you are now in capture mode" moment. The state change is subtle. Similarly, exiting capture mode (either by capturing or by `/scaena tech captureexit`) has no strong confirmation signal.

**Where this breaks:** Fast-moving work sessions where you lose track of what mode you're in.

## Proposed UX Model

The redesign centers on three phases with clear entry points, each with a coherent primary interface.

### The Mental Model: Three Phases

```
DESIGN          →       REHEARSE        →       PERFORM
────────────────────────────────────────────────────────
Configure show          Preview + refine        Run for audience
Mark positions          Toggle departments      /show play
Edit timeline           Test cues               /show stop
Tune parameters         Spatial check           /show list
```

These map to the existing `/scaena` and `/show` namespaces, but with clearer boundaries and entry points.

---

### Phase 1 — The Show Dashboard

**Entry point:** `/scaena <showId>` (shorthand, no subcommand required)

This command loads the prompt-book and renders a **Show Status Dashboard** — a structured, complete picture of where the show stands.

**Dashboard structure (chat panel):**

```
─────────────────────────────────────────────────────
  ✦  showcase.01  ·  Preparing for Battle
  Stage: scouted   Duration target: ~12 min
─────────────────────────────────────────────────────
  SCENES                                  5/6 ready
  [00] Home Base — The Workshop        ✓  scouted
  [01] The Road North                  ✓  scouted
  [02] The Ruined Village              ⚠  pending
  [03] The Fortress Gate               ✓  scouted
  [04] The Great Hall                  ⚠  missing
  [05] Victory Coda                    ✓  scouted
─────────────────────────────────────────────────────
  MARKS                                  11/14 set
  companion_spawn    ✓   vindicator_spawn   ✓
  home_base          ✓   blast_furnace      ⚠ missing
  victory_platform   ⚠  ...
─────────────────────────────────────────────────────
  OPEN ITEMS
  ⬜ victory coda fireworks preset — Mira (pending)
  ⬜ victory levitation amplifier — ceiling clearance
  ⬜ foe_health_multiplier — TBD
─────────────────────────────────────────────────────
  TIMELINE                              12 cues set
  [Browse & Edit Timeline...]
─────────────────────────────────────────────────────
  [Enter Setup →]   [Params]   [Quick-play]
─────────────────────────────────────────────────────
```

Key design decisions:

- **Reads from the prompt-book directly** — readiness.open items, scene status, mark capture status
- **Shows completion fractions** (11/14 marks set) so progress is visible at a glance
- **Timeline row** — shows count of authored cues and links to timeline browser (Phase 2)
- **[Enter Setup →]** — loads the first incomplete scene in tech mode
- **Single command to refresh** — the panel auto-refreshes after any save action; `/scaena <showId>` always shows current state

This is the anchor that Phase 1 is missing. It's not a new data model — all the data exists in the prompt-book — it's a reading surface.

**Implementation path:** New `ShowDashboardBuilder` class (parallels `TechPanelBuilder`). Reads `PromptBook`, `ScoutCapture` files, and show YAML. Renders to chat using existing MiniMessage patterns. Can be extracted from `TechManager.enterTech()` as a standalone query.

### Phase 2 — In-Game Timeline Editor

This is the most significant new feature. It makes the show's cue timeline a first-class in-game object.

**Entry point:** From the dashboard, click `[Browse & Edit Timeline...]`, or via `/scaena <showId> timeline`

**Timeline Panel:**

```
─────────────────────────────────────────────────────
  TIMELINE ·  showcase.01   (18 cues · 1,320 ticks)
─────────────────────────────────────────────────────
  t:000  GROUP_ASSIGN   groups: 2, strategy: random
  t:010  CUE          → intro.player_entrance.warm
  t:060  SOUND          entity.elder_guardian.ambient
  t:100  CUE          → fx.tension_ramp.medium
  t:160  PLAYER_CHOICE  "What path will you take?"
  t:200  CUE          → combat.fanfare.short
  ...
  [+ Add cue at tick...]   [Show cue library]
─────────────────────────────────────────────────────
```

Each timeline entry is **clickable**. Clicking opens a cue detail/edit panel:

```
─────────────────────────────────────────────────────
  CUE  ·  intro.player_entrance.warm  (t:010)
  "Warm entrance ramp — players arrive with fanfare"
  Duration: 170 ticks   Tags: warm, building, ramp
─────────────────────────────────────────────────────
  [Change timing...]   [Replace cue...]   [Remove]
  [Preview this cue]
─────────────────────────────────────────────────────
```

**Cue Library Browser** — the other missing piece:

```
─────────────────────────────────────────────────────
  CUE LIBRARY   (62 cues)   Filter: [all ▾]
─────────────────────────────────────────────────────
  Filter by tag:  [warm] [tense] [combat] [ambient]
                  [fireworks] [player] [sound] [fx]
─────────────────────────────────────────────────────
  intro.player_entrance.warm      warm  building  170t
  intro.player_entrance.tense     tense dark      170t
  fx.tension_ramp.medium          tense building  200t
  fx.tension_ramp.long            tense ramp      350t
  combat.fanfare.short            combat bright   80t
  ...
  [Click any cue to preview or place]
─────────────────────────────────────────────────────
```

Clicking a cue from the library presents:

```
  intro.player_entrance.warm
  "Warm Pulse Ramp — four rounds of warm fireworks..."
  Duration: 170t (8.5s)   Tags: warm, building, ramp
  [Place at tick: ___]   [Preview now]   [Cancel]
```

"Place at tick" accepts a number (or tick → seconds display for legibility). The cue is added to the show's timeline in memory and flagged as unsaved. A `[Save timeline]` action writes it back to the show YAML. This requires a new write path for show timeline YAML — `ShowTimelineWriter` — parallel to `PromptBookWriter`.

**In-place event editing** (for direct events like SOUND, MESSAGE, TITLE — not CUE references):

For simple events on the timeline, clicking should open an **event edit panel** that surfaces the editable fields:

```
  SOUND  ·  t:060
  sound_id:  entity.elder_guardian.ambient
  volume:    1.0
  pitch:     0.8
  category:  master
  [Edit sound_id]   [Edit volume]   [Edit pitch]   [Done]
```

Editing a text field (like sound_id) enters the existing chat-input pattern already used for text params. Editing a numeric field enters the scroll/hotbar pattern already used for NUMERIC params. This reuses the existing param editing infrastructure — no new input paradigms needed.

**Saving:** Timeline edits accumulate in a `modifiedTimeline` structure on the session (parallel to `modifiedParams` and `modifiedMarks`). A `[Save timeline]` button (or `/scaena tech save`) writes the modified show YAML back to disk. A new `ShowTimelineWriter` class handles this.

**Where complex cue editing lives:** Editing the *internal events of a cue* (what `ramp.pulse.warm` does on its own timeline) is a separate authoring concern. For this review, the timeline editor operates at the **show level** (what cues fire when, in the show's timeline), not the **cue level** (what events are inside a cue). Cue-level editing is a future phase; for now, cues are library items that are placed, not authored in-game.

### Phase 3 — Rehearsal Mode (Refinement of Tech)

The existing tech rehearsal mode (`TechSession`, `TechManager`) is largely right in structure. The changes needed are incremental:

**Keep as-is:** Adventure mode, flight, hotbar items, actionbar for capture mode coordinates, sidebar for scene/department status, mark capture flow (slot-8 to set), build mode (SURVIVAL toggle), param scroll mode.

**Change:**

1. **Show all marks in-world by default** — Don't require scout mode to see marker TextDisplays. When entering tech, spawn TextDisplay labels at all mark positions (captured: green, uncaptured: red `?` at best-guess position or origin). Toggle visibility with F-key or `/scaena tech markers`.

2. **Auto-refresh the panel** — After any capture, param change, or department toggle, re-send the main panel. The current panel is sent once on session start and must be manually re-summoned. Auto-refresh eliminates the "stale panel" problem without changing the information.

3. **Inline status on the main panel** — Show capture status symbols on the marks row directly, not only in the separate mark list panel.

4. **Bbox visualization** — When entering build mode, spawn a particle outline of the current bbox (if set). Update it live as corners are adjusted. This gives spatial grounding to the abstract corner coordinates.

5. **Capture mode: stronger signal** — Play a distinctive sound when entering and exiting capture mode (distinct from the confirm sound). Flash the screen border briefly (title event with empty title, or potion effect). The current actionbar-only signal is too subtle.

6. **Dashboard link from tech** — Inside a tech session, a `[Show Overview]` button in the panel links back to the dashboard panel for the show. Easy escape hatch if you need to check overall state mid-session.

---

### Unifying Scout and Tech

Long term: merge `ScoutSession` and `TechSession` into a single `SetupSession`. The distinction between "scouting positions" and "configuring departments" is internal architecture, not a meaningful user boundary.

Near term (minimal disruption): make tech mode absorb scout functionality transparently. When `/scaena tech <showId>` is called and scout_captures exists, load those captures. When a mark is captured in tech mode, write to scout_captures as well as modifiedMarks. Remove the need to ever run a separate `/scaena scout` workflow for a show that already has a prompt-book.

Keep `/scaena scout` operational for initial exploration of a new world/build site where a prompt-book doesn't exist yet. That's the appropriate use case for standalone scouting.

---

### Command Simplification

The existing command surface can be simplified at the entry level without changing the internal structure:

**Today:**
```
/scaena scout load showcase.01 site_a
/scaena tech showcase.01 site_a
/scaena tech panel
/scaena tech params
/scaena tech marklist
```

**Proposed top-level entries:**
```
/scaena <showId>              — Show Dashboard (Phase 1 entry)
/scaena <showId> scene        — Enter setup for a scene (replaces /scaena tech)
/scaena <showId> timeline     — Open timeline editor (Phase 2)
/scaena list                  — Browse all shows with status
/scaena choose <index>        — PLAYER_CHOICE (unchanged)
```

All the detailed subcommands (`toggle`, `capture`, `focusparam`, etc.) remain as explicit commands inside an active session but are also accessible via panel buttons — the commands become fallbacks for keyboard-first users, not the primary navigation method.

## Architecture Implications

### New Components Required

**`ShowDashboardBuilder`** — Reads PromptBook + scout capture files + loaded Show, renders a structured status panel. Stateless query — can run without an active session.

**`TimelineEditorSession`** (or extend `TechSession`) — Holds in-memory edits to the show's cue timeline. Tracks `addedCues`, `removedCues`, `rescheduledCues` as delta lists. On save, applies delta to show YAML.

**`ShowTimelineWriter`** — Writes the modified timeline back to the show YAML file. Parallel to `PromptBookWriter`. Needs careful YAML preservation (don't reformat or reorder non-timeline fields). Should produce a diff log for the revision log.

**`CueLibraryBrowser`** — Renders the cue library as a browsable, filterable chat panel. Reads from `CueRegistry`. Supports tag filtering (which maps to the existing `tags:` field on `Cue`).

**`MarkVisualizationTask`** — Manages in-world TextDisplay entities for all marks in the current scene. Color-coded by capture status. Runs in tech mode by default (not only in scout mode). Replaces duplicate logic in `ScoutManager` and `TechManager`.

### Existing Components to Modify

**`TechPanelBuilder`** — Add capture status symbols to marks row. Add auto-refresh trigger. Add `[Show Overview]` and `[Timeline]` buttons. Evolve into `ShowEditorPanelBuilder` (or keep name, expand scope).

**`TechSession`** — Add `modifiedTimeline` delta list. Add `marksVisualization` task reference. Slightly expanded state, no structural change.

**`TechManager`** — Add `loadMarkVisualizations()` call in `loadScene()`. Add auto-refresh of panel after state-changing operations. Add dashboard query path that doesn't require entering a full session.

**`ScoutManager` / `ScoutSession`** — No changes required near-term if tech absorbs scout functionality. Long-term: merge into `SetupManager`.

### What the Architecture Already Supports

The existing param editing system (`ParamSpec` with type/step/min/max, scroll mode in actionbar, chat-input for text params) is exactly the right pattern to reuse for inline event field editing in the timeline editor. No new input paradigm needed — the timeline editor can piggyback on the same infrastructure.

The `PromptBookWriter` pattern is a clean model for `ShowTimelineWriter`. Write specific sections back to YAML, preserve the rest.

The `TechHotbarListener` (F-key hotbar swaps) is the right hook for "confirm capture" — already in place.

## Priority Levels

### Fix Immediately

**1. Show Status Dashboard** (`/scaena <showId>`)
One command that shows complete show state: scenes, marks, open items, timeline cue count. Reads existing data. No new data model. Highest ROI for lowest implementation cost.

**2. Timeline visibility** — even read-only
A `/scaena <showId> timeline` command that lists the current show timeline (cues, timing, event types) as a chat panel. Even without editing capability, seeing the timeline in-game is valuable. The edit layer can come second.

**3. In-world marks in tech mode**
`MarkVisualizationTask` — TextDisplay markers for all marks in the current scene, visible by default when entering tech. Currently only available in scout mode. High spatial value for low implementation cost (the TextDisplay spawning code already exists in `ScoutManager`).

---

### Improve Next

**4. Timeline editing** — add + remove cues
`TimelineEditorSession` + `ShowTimelineWriter`. The ability to add a cue from the library at a given tick, or remove an existing one, closes the "must edit YAML" loop for the most common authoring operations.

**5. Cue library browser**
`CueLibraryBrowser` with tag filtering. Browse 60+ cues in-game, see description and duration, place on timeline. Turns the existing cue library from an internal asset into a usable design tool.

**6. Panel auto-refresh**
After any capture, param edit, or department toggle, automatically re-send the main panel. Eliminates the single most common source of stale UI.

**7. Absorb scout into tech**
Make `/scaena tech` transparently load prior captures and allow writing new ones to the same files. Eliminate the need for a separate scout session for shows that already have a prompt-book.

---

### Nice to Have

**8. Inline event field editing**
Extend the timeline panel so clicking a direct event (SOUND, MESSAGE, etc.) opens a field editor using the existing param editing infrastructure. Allows tuning sound pitch, particle count, message text without touching YAML.

**9. Bbox particle visualization**
Particle outline of the current bounding box when in build mode. Live update as corners are adjusted.

**10. Guided first-session flow**
When opening a show with no captures yet, walk through "Step 1: set your anchor mark. Step 2: set scene entry points. Step 3: review casting." Useful for onboarding Smitty to a new show without Alan present.

**11. Inventory GUI for show selection**
`/scaena list` opens a chest GUI with shows as items, status shown in item lore. More discoverable than a flat chat list, especially for a second admin.

**12. Cue preview mode**
From the timeline or library browser, `/scaena preview <cueId>` — fires the cue immediately for the current player, without it being part of a running show. Fast way to evaluate a cue's effect in the actual build location.
